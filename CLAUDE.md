# CLAUDE.md

이 파일은 Claude Code가 이 Obsidian vault에서 일할 때 따르는 **업무 규약**입니다. (다른 에이전트는 [[AGENTS]]도 함께 봅니다.)

목표는 개인 메모장이 아니라, 여러 AI 에이전트와 사람이 **같은 업무 맥락을 공유**하는 안정적인 비즈니스 프로세스입니다.
(설계 출발점: Karpathy LLM Wiki — raw는 불변, wiki는 에이전트가 키우는 가공 지식, 탐색은 복리로 쌓는다.)

## 0. 세션 루틴

- **세션 시작**: `query`(="옵시디언 참조")로 [[index]] · [[log]] · 관련 wiki 문서를 먼저 읽고 맥락을 복원한다.
- **세션 종료**: `save`(="이번 작업 내용 옵시디언에 저장해줘")로 저장 가치가 있는 것만 저장한다.
- 시작 시 `tasks/lessons.md`(있다면)와 관련 [[가이드라인 개요|가이드라인]]을 검토한다.

## 1. Core Operating Rules

1. 작업 전 [[index]], [[log]], 관련 `AI-Sessions/wiki/` 문서를 먼저 확인한다.
2. `AI-Sessions/raw/`의 원본은 **수정·삭제하지 않는다**(읽기 전용).
3. 가공된 지식·결정·에러·프로젝트 문서는 `AI-Sessions/wiki/` 아래에 저장한다.
4. 세션 인수인계는 `AI-Sessions/conversations/`에 저장한다.
5. 중요한 저장 후 [[index]]와 [[log]]를 갱신한다.
6. 사용자가 명시적으로 원하지 않는 한 민감정보·토큰·비밀번호·고객 개인정보를 저장하지 않는다.

## 2. Korean / English Hybrid (언어 정책)

- **사람이 읽고 편집하는 가이드라인 = 한국어.** 톤앤매너·금지표현·나레이션·기획/리뷰 기준 등. → [[가이드라인 개요]]
- **에이전트 실행 키워드 = 영어 고정.** `save` · `ingest` · `query` · `lint`. 의미·철자를 바꾸지 않는다.
- 아래 §3의 자연어 트리거도 항상 같은 키워드로 해석한다.

## 3. 명령어 계약 (Command Contract)

각 키워드가 **언제, 무엇을 읽고, 무엇을 쓰는지**를 고정한다. 이 표가 read/write의 **단일 기준**이다. (사람용 요약은 [[명령어_사용법]].)

| 키워드 | 자연어 트리거 | READ (읽기) | WRITE (쓰기) | 필수 안전장치 |
|---|---|---|---|---|
| `query` | "옵시디언 참조", "옵시디언 참조해줘" | [[index]], [[log]], 관련 `wiki/**` | 기본 없음. 가치 있는 종합은 §6에 따라 `save`로 승격(쓰기는 save 계약을 따름) | raw 미수정. 출처 문서를 함께 인용 |
| `save` | "이번 작업 내용 옵시디언에 저장해줘", "옵시디언에 저장" | 저장 대상 맥락 + 중복 확인용 관련 `wiki/**` | 해당 `wiki/<카테고리>/` 문서(신규 또는 갱신) + [[index]] + [[log]] | §5 저장 필터 5개 + §5 안전장치 |
| `ingest` | "이 자료 ingest 해줘", "raw 가공해줘" | `AI-Sessions/raw/**` (읽기 전용) | `wiki/sources/` 요약 + 관련 `concepts/`·`decisions/` + [[index]] + [[log]] | raw 원본 절대 미수정. 출처 경로 명시 |
| `lint` | "위키 lint", "위키 점검해줘" | vault 전체(`wiki/**`, [[index]], [[log]], raw 경계) | 점검 리포트(채팅). 실제 수정은 사용자 승인 후 | 자동 삭제 금지. 제안 먼저 |

## 4. Raw / Wiki 분리 (불변 원칙)

- `AI-Sessions/raw/` = **불변 자료**. 기사 원문, 회의·통화 녹취, 외부 문서, 화면 캡처 설명 등 근거로 다시 볼 자료.
- 에이전트는 raw를 **수정하지 않는다.** raw를 바탕으로 요약·판단·결정·기획을 만들 땐 반드시 `wiki/` 아래 **새 문서**로 만든다.
- `conversations/`는 인수인계용. 검증 안 된 중간 생각은 여기까지만 두고 wiki로 **승격하지 않는다.**

## 5. Save Filter & Safeguards (맥락 오염 방지)

무분별한 저장은 맥락을 오염시킨다. `save` 전에 **반드시** 통과해야 한다.

### 5.1 저장 필터 (5개 중 하나 이상 — 아니면 저장 안 함)

1. 향후 실무에 **반복 재사용**될 데이터인가?
2. 다른 에이전트·동료가 프로젝트를 **이어받기 위해 반드시 읽어야** 하는가?
3. 의사결정의 **근거와 결정권자**를 나중에 추적할 필요가 있는가?
4. **실패한 방식**이라 다시 시도하면 안 되는 리스크 정보인가?
5. 팀이 맞춰야 할 **공통 규칙·디자인 가이드**인가?

하나도 해당하지 않는 **일회성 답변·감상·사소한 표현 변경·검증 안 된 추측**은 wiki에 저장하지 않는다. 필요하면 `conversations/`에만 남기고, **저장하지 않은 이유를 짧게 보고**한다.

### 5.2 저장 안전장치 (필터 통과 후에도)

- **중복 확인**: 저장 전 같은 주제 문서가 있는지 확인하고, 있으면 새로 만들지 말고 **갱신**한다.
- **출처 명기**: 결정·주장은 근거(출처/대화/raw 경로)와 날짜를 함께 적는다. **출처 없는 단정 금지.**
- **검증 상태 표시**: 확정 전이면 `status: draft`, 검증 후 `active`, 대체되면 `superseded`.
- **결정권자 표기**: 결정·가이드라인은 `owner`(기본: 원장)를 적는다.
- **raw 불변**: 저장이 raw를 건드리려 하면 **중단**한다.
- **민감정보 차단**: 토큰·비밀번호·개인정보는 본문에 쓰지 않고 위치만 참조한다. → [[금지표현]]
- **추측 분리**: 추측은 추측으로 표시. 단정처럼 쓰지 않는다.

## 6. Query Compounding (탐색의 복리)

`query`로 만든 가치 있는 종합(비교표·분석·정리)은 채팅에 흘려보내지 말고, §5를 통과하면 wiki 새 문서로 `save`한다. 탐색이 누적되어야 위키가 복리로 성장한다. (Karpathy)

## 7. Wiki Categories

- `wiki/sources/` — raw 요약·출처 맥락
- `wiki/concepts/` — 반복 개념·용어·프레임워크
- `wiki/decisions/` — 의사결정·근거·결정권자·날짜
- `wiki/errors/` — 실패·재시도 금지 리스크
- `wiki/projects/` — 프로젝트별 진행 맥락·산출물
- `wiki/design/` — 디자인·브랜드·IA → [[브랜드_디자인_가이드]]
- `wiki/dev-tasks/` — 개발 작업 단위·의존성·구현 메모
- `wiki/guidelines/` — 사람이 읽는 판단·품질 기준(한국어) → [[가이드라인 개요]]

에이닷 영어학원 운영 전용:

- `wiki/departments/` — 부서(에이전트 역할). 시작점 [[부서 개요]]
- `wiki/sops/` — 업무 표준 절차
- `wiki/students/` — 학생 프로필(새 학생은 `templates/템플릿_학생` 복사)

기타: `templates/`(새 문서 템플릿), `tasks/`(`todo.md` 계획·리뷰, `lessons.md` 교정 패턴), `prompts/`(명령 프롬프트).

## 8. Document Format

```markdown
---
type: decision | source | concept | error | project | design | dev-task | guideline | handoff | department | sop | student
date: YYYY-MM-DD
status: draft | active | superseded
owner: 결정권자 (decision·guideline 권장, 기본 원장)
source: 출처/raw 경로 (선택)
review: 재검토 예정일 (선택)
---

# 제목

## Summary
## Context
## Details
## Links
```

## 9. Lint Checklist

`lint` 실행 시 점검한다(자동 삭제 금지, 제안 먼저).

1. **검증 안 된 추측**이 단정처럼 wiki에 있는가 (→ `status: draft` 또는 conversations로 강등)
2. **중복 정보** — 같은 내용이 여러 문서에 흩어져 있는가 (→ 한 곳으로 통합)
3. **출처 없는 결정** — 근거·결정권자 없는 주장이 있는가 (→ 출처 보강 또는 강등)
4. **낡은 규칙(stale)** — 새 자료로 대체된 내용이 남아 있는가 (→ `status: superseded`)
5. 문서 간 **모순**되는 주장
6. 인바운드 링크 없는 **고아 문서**, 누락된 상호 링크
7. 자주 언급되나 **자체 문서 없는 개념**
8. raw 원본이 수정되었거나 wiki에 섞였는가 / 민감정보가 저장됐는가
9. [[index]] · [[log]] 누락 항목

## 10. Workflow Orchestration

1. **Plan First** — 3단계+ 작업은 `tasks/todo.md`에 체크 가능한 계획. 막히면 멈추고 재계획.
2. **Subagent 활용** — 리서치·탐색·병렬 분석은 서브에이전트로. 하나당 하나의 접근.
3. **Self-Improvement Loop** — 교정받으면 즉시 `tasks/lessons.md`에 패턴 기록, 세션 시작 시 검토.
4. **Verification Before Done** — 작동 증명 없이 완료 처리 금지. → [[리뷰기준]]
5. **Demand Elegance** — 사소하지 않은 변경은 "더 우아한 방법?" 자문. 단순 수정엔 과잉설계 금지.
6. **Autonomous Fixing** — 버그는 근본 원인까지 직접. 임시방편 금지.
7. **Simplicity First** — 최소 영향. 필요한 것만.

## 11. Completion Rule

작업 종료 시 보고한다.

- 읽은 주요 파일
- 생성·수정한 파일
- 저장 필터 적용 결과(저장/미저장 + 이유)
- 다음 세션에서 먼저 볼 문서
