# 구글시트 타임존 오설정 → 날짜가 하루 밀려 보임

## Summary

단어시험 기록이 **아직 오지 않은 날짜**로 관제에 떴다(실제 7/20 18:02 → 화면엔 **7/21 10:02**). 데이터는 멀쩡했고, 원인은 **스프레드시트의 타임존이 `America/Los_Angeles`로 설정**되어 있던 것. 서울로 바꿔 **전 기록이 한 번에 교정**됨.

## Context

- 대상 시트: 단어테스트기록(상균T 스프레드시트). 백엔드는 `Utilities.formatDate(new Date(), 'Asia/Seoul', 'yyyy-MM-dd HH:mm')`로 **한국시간 문자열을 올바르게 기록**하고 있었다.
- 증상: 원장이 "7월 21일이 되지도 않았는데 왜 21일로 뜨냐"고 발견.

## Details

### 왜 밀렸나

1. 백엔드가 `"2026-07-20 18:02"` 문자열을 기록 → 시트가 **날짜값(serial)으로 자동 변환**.
2. Apps Script가 그 셀을 읽으면 **시트 타임존(LA)** 기준으로 Date 객체를 만들고, **스크립트 타임존(KST)** 으로 문자열화한다.
3. LA(UTC-7) → 서울(UTC+9) = **+16시간** → 7/20 18:02 이 **7/21 10:02** 로 출력. 관측된 밀림과 정확히 일치.

### 진단이 헷갈렸던 이유

- **gspread로 읽으면 정상**("2026-07-20 18:02")인데 **Apps Script로 읽으면 밀린 값**이 나왔다. 같은 셀인데 읽는 경로에 따라 달랐던 것이 결정적 단서.
- 로컬 PC 시계·구글 서버 시계(HTTP Date 헤더)를 대조해 **현재 시각은 정상**임을 먼저 확정한 뒤에야 원인이 좁혀졌다.

### 해결

- 스프레드시트 속성 `timeZone`을 **`Asia/Seoul`** 로 변경(Sheets API `updateSpreadsheetProperties`).
- 시트에 저장된 serial은 **벽시계값**이라 타임존만 맞추면 **기존 기록까지 전부 정상 표시**된다(데이터 수정 불필요).

## 재시도 금지 / 체크리스트

- ❌ 미래 날짜가 보인다고 **기록을 고치거나 지우지 말 것**. 데이터가 아니라 표시 설정 문제일 수 있다.
- ✅ 날짜가 이상하면 먼저 **① 현재 시각(로컬 vs 구글 헤더) ② 읽는 경로별 값 비교(gspread vs Apps Script) ③ 시트 timeZone** 순으로 확인.
- ✅ **새 스프레드시트를 만들면 타임존을 Asia/Seoul로 지정**한다(기본값이 미국일 수 있음).
- 참고: 이후 관제는 단어시험을 exam_server가 **시트 원본 문자열 그대로** 읽어 쓰므로(`/api/wordtests`) 이 변환 경로를 아예 타지 않는다.
